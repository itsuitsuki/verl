Below are the reasoning steps of a model's solution. Each step lists premises and exactly one conclusion. For EACH step, translate its stated premises AND its conclusion into Python boolean expressions. Output ONLY the STEP lines, one per input step, same numbering, nothing else - no explanations, no markdown:

Rules:
- premises: the facts THIS step's premise lines state, separated by ' ; ' (write - if it states none). Translate them as written; they must come from the problem or from earlier steps, never from this step's own conclusion.
- prop: the conclusion as ONE Python boolean expression.
- Python subset: == for equality, explicit *, ** powers, / real division, // integer division, % modulo, and/or to combine facts. Allowed functions only (sqrt abs gcd lcm floor ceil factorial comb log ln exp sin cos tan min max prime guarantee_pair guarantee_same); prime(n) is True exactly when the integer n is prime. guarantee_pair(n) is the fewest draws over n categories that force a matching pair (= the certified pigeonhole minimum); guarantee_same(n, k) forces k in one category -- use these for "minimum to guarantee ..." conclusions instead of the expanded n+1 / n*(k-1)+1 formula. The mathematical constants are pi (write pi for the circle constant, e.g. a right angle is pi/2, and tan(pi - x), sin(pi/2 - x) simplify) and Euler's number written exp(1); never write a bare e for Euler's number (e reads as a variable). A finite sum or product is written sum(<expr> for k in range(lo, hi)) or prod(<expr> for k in range(lo, hi)) -- this generator is the ONLY comprehension allowed (exactly one 'for', no 'if', index k runs lo, lo+1, ..., hi-1). A fact holding for all / some value uses forall(x, <body>) / exists(x, <body>) (x is the bound variable). Otherwise a_n is a plain variable, never a(n); no sets, tuples, subscripts, or non-sum comprehensions.
- DEFINED FUNCTION: if a premise (here or earlier) DEFINES a function by a formula, e.g. f(x) = 3*x**3 - x + a, then your translated prop MUST NOT contain any f(...) term at all -- INLINE every f(<arg>) by substituting <arg> for the bound variable in the formula. Examples: with f(x) = 3*x**3 - x + a, translate the conclusion "f(2) = 22 + a" as 3*2**3 - 2 + a == 22 + a; translate "f(2) = 0" as 3*2**3 - 2 + a == 0; translate "f(3) = 3**2 + 1 = 10" (from f(x)=x**2+1) as 3**2 + 1 == 10. NEVER keep f(2) as a symbol and NEVER introduce a variable like f_2 or f2 for a defined function's value -- either loses the definition and the step cannot be verified. (An UNDEFINED function -- one only constrained by an equation it satisfies, with no closed formula -- has no formula to inline; keep it as a FUNCTION named f, g, or h and APPLY it: write f(2), f(x), f(x+1), f(log(b, x)) directly. Translate a law that holds for ALL inputs with forall: "f(x+y) = f(x) + f(y) for all x, y" -> forall(x, forall(y, f(x+y) == f(x) + f(y))); "f(log(b, x)) = x for all x > 0" -> forall(x, implies(x > 0, f(log(b, x)) == x)) -- use implies(condition, body) for a guarded "for all ... such that". Note log(base, value): log base 10 of z is log(10, z), log base 2 of 8 is log(2, 8) (which equals 3). An instantiated step stays applied: "f(2) = f(1) + f(1)" -> f(2) == f(1) + f(1). Only f, g, h may be undefined functions.)
- Standard forms for common topics: "x is divisible by d" -> x % d == 0; "the remainder of x divided by m is r" -> x % m == r; "a and b are congruent mod m" -> a % m == b % m; "p is prime" -> prime(p). The LCM or GCD of a run of integers a through b is a NESTED fold written with the literal endpoints: "the LCM of the integers 1 through 4" -> lcm(1, lcm(2, lcm(3, 4))) and "the GCD of 6 through 9" -> gcd(6, gcd(7, gcd(8, 9))); put the SAME fold in both the premise and the conclusion and NEVER give it a variable name, so "the LCM of 1 through 9 is 2520" -> lcm(1, lcm(2, lcm(3, lcm(4, lcm(5, lcm(6, lcm(7, lcm(8, 9)))))))) == 2520 (a concrete fold both sides, verifiable by evaluation). Factoring or polynomial division: the check IS the identity -- write the ORIGINAL expression == the factored / simplified form as ONE equation (factoring: 58*x**5 - 203*x**11 == 29*x**5*(2 - 7*x**6)), which Isabelle verifies by expanding both sides; a wrong factoring makes the two sides differ and correctly fails. For a DIVISION the identity only holds where the divisor is nonzero, so ADD the divisor != 0 as a premise: premise 2*x*y != 0, prop (6*x**2*y - 2*x*y**2)/(2*x*y) == 3*x - y. Do NOT use gcd() or lcm() on expressions that contain a VARIABLE (they are for integers only): a sub-premise like "the GCF of x^5 and x^11 is x^5" is NOT translatable -- drop it, the identity alone proves the factoring. A discrete probability -> favorable / total, counting cases with comb (e.g. comb(5, 2)). Coordinate geometry: give each point two coordinate variables, prefer squared distance (x2-x1)**2 + (y2-y1)**2 over sqrt, and write a line or circle as its defining equation. A number sequence: when a premise says the sequence is arithmetic or geometric, you MUST add its defining relations as premises (otherwise the later terms are unconstrained and the step cannot be verified) -- for an arithmetic sequence use a fresh common difference d and write a_2 == a_1 + d ; a_3 == a_1 + 2*d ; a_5 == a_1 + 4*d ; for a geometric sequence use a fresh ratio r and write a_2 == a_1*r ; a_3 == a_1*r**2. The terms, d and r are REAL (the difference/ratio can be fractional even when the terms look integer; an int declaration makes the relations contradictory). When a step evaluates a finite sum or series to a number, write sum(...) == <value> with both sides concrete (not answer == sum(...)).
- NEGATION of a QUANTIFIED proposition: a step concluding the negation of a quantified statement ("the negation of 'for all x in D, B(x)' is 'there exists x in D, not B(x)'") is checked as a logical EQUIVALENCE -- write the prop as not(<original proposition>) == <stated negation>. Formalize "for all x in D, B" as forall(x, implies(<D as a condition on x>, B)) and "there exists x in D, B" as exists(x, (<D condition>) and (B)); "x in (0, +inf)" is the condition x > 0, "x in R" / "for all real x" drops the guard. The correct negation swaps forall<->exists and negates the inner relation (>= becomes <, > becomes <=, == becomes !=, and vice versa); a WRONG negation makes the equivalence false and correctly fails. Example: "the negation of 'for all x > 0, x^2 + a*x + 3 >= 0' is 'there exists x > 0, x^2 + a*x + 3 < 0'" -> not(forall(x, implies(x > 0, x**2 + a*x + 3 >= 0))) == exists(x, (x > 0) and (x**2 + a*x + 3 < 0)).
- Matrices and vectors: expand them into scalar components with the ordinary operations -- never use matrix/vector notation. A 2x2 determinant of [[a,b],[c,d]] -> a*d - b*c ; a 3x3 determinant by cofactor expansion; a dot product (a1,a2,a3).(b1,b2,b3) -> a1*b1 + a2*b2 + a3*b3 ; a squared magnitude |v|**2 -> v1**2 + v2**2 + v3**2 ; a matrix product entry -> the sum of the products of the row and column components.
- Calculus: for an expression e written in the variable x -- a derivative "the derivative of e at p is d" -> deriv(e, x, p, d) (e.g. f'(3)=6 for f(x)=x**2 -> deriv(x**2, x, 3, 6)); a definite integral from a to b, integral of e dx -> integral(e, x, a, b), so "the integral of 2x from 0 to 1 is 1" -> integral(2*x, x, 0, 1) == 1; a limit "the limit of e as x approaches p is L" -> limit(e, x, p, L), so "lim x->0 of sin(x)/x is 1" -> limit(sin(x)/x, x, 0, 1).
- Be faithful to the text AS WRITTEN: do not correct arithmetic, do not invent numbers. Every number the conclusion states must appear in prop doing real work - never satisfy this with tautologies (5 == 5) or by restating a given. Do NOT PRE-SIMPLIFY a numeric computation the step writes out: if the step shows "2 + 1 = 2x - x", translate the sides AS WRITTEN, 2 + 1 == 2*x - x, NOT the collapsed 3 == x -- the written form 2 + 1 is verified from its parts and keeps 3 traceable to the 2 and 1 the step states, whereas a bare 3 the step never literally wrote reads as an invented number.
- Use the declared variables; introduce fresh lowercase names for new quantities and reuse them consistently across steps. Keep each name's meaning FIXED: when the problem has several objects of the same kind -- two triangles, an original figure and a derived one, a before state and an after state, two circles -- a name that stands for one object's quantity must NEVER be reused for another object's. Give the second object its own distinct name (e.g. a_orig vs a_new, area1 vs area2). Reusing a name for two different quantities makes the premises contradict each other (side == sin A in one place and side == 2*sin A in another forces sin A == 0), and the whole step fails as inconsistent.
- When the conclusion names an aggregate or derived quantity (total, sum, remaining, difference, average, probability, product) computed from quantities in this step's premises, write its defining relation explicitly as a conjunct: total_cost == house_cost + repair_cost and total_cost == 130000. The word's meaning is part of the conclusion.
- `answer == <value>` is ONLY for the value that DIRECTLY answers the QUESTION. A step concluding an INTERMEDIATE amount that a LATER step still builds on -- "the pool currently has 80 tiles" when the question asks for RED tiles, "there are 46 apples total" before some are sold, a running subtotal -- must get a FRESH lowercase name bound to its value (current_tiles == 80, total_apples == 46), NEVER `answer`. Mapping an intermediate onto `answer` contradicts the real answer (answer == 80 while answer is really 32) and makes the premises inconsistent, poisoning every later step. Only the step that states the final asked quantity uses `answer`.
- If a step concludes a bare final value, write answer == <value>. But when it COMPUTES a named quantity from the premises' variables -- "the probability is p*q*r = 25/189", "the area is AE*BF = 750" -- give that quantity a FRESH lowercase name (prob, area, ...) and bind it to the EXPRESSION IN THOSE VARIABLES and the value in ONE prop: prob == p*q*r and prob == 25/189 (with premises p==1/3, q==2/7-... ). Do NOT use `answer` for this (answer cannot be defined by an expression) and do NOT substitute the numbers into the expression (write area == AE*BF, never area == 30*25) -- the variable form is what lets the value be verified from the premises.
